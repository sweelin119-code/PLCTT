# 房屋、车位与业主管理模块设计方案

> 版本：v2.0 | 日期：2026-04-29 | 状态：待确认

---

## 一、项目背景

当前系统的组织架构已支持到 **小区/项目（project）** 级别，但小区内部的**楼栋、单元、房屋、车位、业主**等基础资产数据尚未建立。这些数据是物业管理的根基：

- **收费管理**：账单需要按房屋生成，收费标准需要按房屋类型/面积定价
- **报修工单**：需要精确到楼栋单元房号
- **业主管理**：业主信息维护、业主与房屋的绑定关系、业主变更追溯
- **车位管理**：车位归属与收费
- **设备管理**：设备安装在具体楼栋/位置
- **门禁管理**：按房屋授权通行权限

本方案参考了市场上主流物业管理系统（如：**千丁云物业、思源物业、海纳云、金蝶我家云**等）的资产管理设计思路，结合本系统**政府→街道→物业→业主**的全链路架构特点，制定一套**完整、规范、可扩展**的小区基础资产管理方案。

## 二、功能范围与边界

### In Scope（范围内）

| 模块 | 功能点 | 优先级 |
|------|--------|--------|
| **楼栋管理** | 楼栋新增/编辑/删除、楼栋信息维护（层数/户数/电梯数/建成年份） | P0 |
| **单元管理** | 单元新增/编辑/删除、单元信息维护 | P0 |
| **房屋管理** | 房屋新增/编辑/删除、房屋信息（面积/户型/朝向/装修状态）、批量生成（按楼层模板） | P0 |
| **业主管理** | 业主档案管理、业主列表/搜索、业主分组标签、业主变更历史追溯 | P0 |
| **业主绑定** | 房屋与业主的绑定/解绑、多业主/共有产权、家庭成员/租户管理 | P0 |
| **车位管理** | 车位新增/编辑/删除、车位类型（固定/临时/机械）、车位区域划分 | P0 |
| **车位绑定** | 车位与业主绑定、车位与房屋关联、车位租赁管理 | P1 |
| **政府端数据同步** | 住建局数据对接、资质认证后自动同步、手动触发同步、同步日志 | P0 |
| **物业端多小区切换** | 物业管理端右上角小区切换器、当前小区上下文、数据按小区隔离 | P0 |
| **楼栋/房屋可视化** | 楼栋2D视图、房屋状态着色（已售/已租/空置/装修中） | P2 |
| **数据导入导出** | Excel批量导入楼栋/房屋/车位/业主、导出资产台账 | P1 |

### Out of Scope（范围外）

- 房屋三维建模/BIM展示（后续可视化大屏实现）
- 房屋销售/租赁交易流程（仅管理归属关系）
- 车位临停计费（归入收费管理模块）
- 房屋维修历史（归入设备/工单模块）

### 3.2 数据来源标识（dataSource）

每条房屋、车位、业主数据都需要标记来源，以支持政府端数据同步：

| 数据来源 | 标识值 | 说明 |
|----------|--------|------|
| 物业手动录入 | `manual` | 物业人员在物业管理端手动录入 |
| 政府端同步 | `gov_sync` | 从住建局/政府系统自动同步 |
| Excel导入 | `import` | 通过Excel批量导入 |
| 业主自主登记 | `owner_register` | 业主在微信端自主登记 |

### 3.4 车位类型与状态

**车位类型：**
| 类型 | 说明 | 计费方式 |
|------|------|----------|
| 固定车位 | 业主购买/长租，固定位置 | 月租/年租 |
| 临时车位 | 公共区域，先到先停 | 按时计费 |
| 机械车位 | 立体机械车库 | 月租/次 |
| 子母车位 | 一个车位可停两辆车 | 月租 |

**车位状态：**
```
vacant(空置) → occupied(已占用) → reserved(已预订) → maintenance(维护中)
```

### 3.5 业主与房屋的关系模型

支持多种绑定关系：

| 关系类型 | 说明 | 权限 |
|----------|------|------|
| **业主（owner）** | 房产证持有人 | 全部权限（投票/出租/出售） |
| **共有产权人（co_owner）** | 房产证上的共有人 | 与业主相同 |
| **家庭成员（family）** | 同住的配偶/子女/父母 | 基本服务（报修/缴费/门禁） |
| **租户（tenant）** | 租赁居住 | 有限服务（报修/缴费） |

### 3.6 业主档案模型

业主档案独立于房屋绑定，支持一个业主拥有多套房屋：

| 字段 | 说明 |
|------|------|
| 姓名 | 业主真实姓名 |
| 手机号 | 登录账号/联系方式 |
| 身份证号 | 实名认证 |
| 性别/生日 | 基础信息 |
| 民族/籍贯 | 人口统计 |
| 学历/职业 | 人口统计 |
| 微信OpenID | 微信端绑定 |
| 标签 | 业主分组标签（如：党员/志愿者/高龄/重点关怀） |
| 名下房屋数 | 自动统计 |
| 名下车辆数 | 自动统计 |
| 数据来源 | manual/gov_sync/import/owner_register |

## 四、业务流程

### 4.1 小区资产初始化流程（物业端手动录入）

```
┌─────────────────────────────────────────────────────────────────┐
│  物业公司入驻 → 创建小区项目（已有）                                │
└──────────────────────────┬──────────────────────────────────────┘
                           ▼
┌─────────────────────────────────────────────────────────────────┐
│  第一步：创建楼栋                                                  │
│  → 录入楼栋名称、层数、单元数、电梯数、建成年份                      │
│  → 支持批量创建（如：1栋~10栋）                                    │
└──────────────────────────┬──────────────────────────────────────┘
                           ▼
┌─────────────────────────────────────────────────────────────────┐
│  第二步：创建单元（可选，无单元的小区可跳过）                        │
│  → 录入单元名称、楼层数                                            │
└──────────────────────────┬──────────────────────────────────────┘
                           ▼
┌─────────────────────────────────────────────────────────────────┐
│  第三步：批量生成房屋                                              │
│  → 选择楼栋/单元 → 设置楼层范围 → 设置每层户数                      │
│  → 设置房屋模板（面积/户型/朝向） → 一键批量生成                    │
│  → 支持Excel导入                                                  │
└──────────────────────────┬──────────────────────────────────────┘
                           ▼
┌─────────────────────────────────────────────────────────────────┐
│  第四步：创建车位                                                  │
│  → 选择区域（地上/地下/机械车库） → 批量生成车位编号                 │

│  第五步：业主管理/绑定                                             │

```

### 4.2 政府端数据同步流程（新增）

```
┌─────────────────────────────────────────────────────────────────┐
│  住建局/政府系统（数据源）                                          │
│  → 楼盘表数据（楼栋/单元/房屋）                                     │
│  → 产权登记数据（业主/产权人）                                      │
│  → 规划审批数据（车位配比）                                         │
└──────────────────────────┬──────────────────────────────────────┘
                           │ 数据推送/接口对接
                           ▼
┌─────────────────────────────────────────────────────────────────┐
│  政府监管端（数据中转）                                              │
│                                                                  │
│  1. 物业企业提交资质认证申请                                        │
│  2. 政府审核通过 → 物业企业获得资质认证                              │
│  3. 物业企业申请管理某小区 → 政府审批通过                            │
│  4. 审批通过后 → 触发数据同步                                       │
│     ├─ 自动同步：系统自动从住建局接口拉取该小区楼栋/房屋/业主数据      │
│     └─ 手动同步：政府人员可在后台手动触发同步                         │
└──────────────────────────┬──────────────────────────────────────┘
                           │ 同步数据
                           ▼
┌─────────────────────────────────────────────────────────────────┐
│  物业管理端（接收数据）                                              │
│                                                                  │
│  1. 收到同步数据 → 写入本地资产库（dataSource='gov_sync'）          │
│  2. 如有冲突（已存在手动录入数据）→ 标记冲突，人工确认                │
│  3. 同步完成后 → 物业人员可查看/编辑/补充数据                        │
│  4. 同步日志可追溯（谁/何时/同步了哪些数据）                         │
└─────────────────────────────────────────────────────────────────┘
```

**同步触发时机：**

| 触发方式 | 说明 | 自动/手动 |
|----------|------|-----------|
| 资质认证通过 | 物业企业资质审核通过后，自动同步其管理小区的数据 | 自动 |
| 小区管理获批 | 物业企业获批管理新小区后，自动同步该小区数据 | 自动 |
| 定时同步 | 每日凌晨自动同步增量数据（产权变更/新楼盘交付） | 自动 |
| 手动触发 | 政府人员在政府端手动点击"同步数据" | 手动 |
| 物业端请求 | 物业人员在物业管理端点击"请求同步" | 手动 |

**同步内容范围：**

| 数据类型 | 同步内容 | 更新策略 |
|----------|----------|----------|
| 楼栋 | 楼栋名称、层数、单元数、建成年份 | 以政府端为准，物业端可补充 |
| 房屋 | 房号、面积、户型、朝向 | 以政府端为准，物业端可补充 |
| 业主 | 姓名、身份证号、手机号（脱敏） | 以政府端为准，物业端可补充联系方式 |
| 产权关系 | 房屋与业主的绑定关系 | 以政府端为准 |

│ 原业主    │ → │ 物业核实  │ → │ 解绑原    │ → │ 绑定新    │
│ 申请变更  │    │ 资料      │    │ 业主      │    │ 业主      │
└──────────┘    └──────────┘    └──────────┘    └──────────┘
                                                              │
                    ┌─────────────────────────────────────────┘
                    ▼
┌─────────────────────────────────────────────────────────────────┐
│  变更记录归档：                                                    │
│  → 记录变更时间、操作人、原业主、新业主                             │
│  → 历史账单关联原业主，新账单关联新业主                             │
│  → 同步通知政府端更新产权信息（可选）                               │
└─────────────────────────────────────────────────────────────────┘
```

**小区切换器设计要点：**
1. 位置：Header 右侧，标题与通知铃铛之间
2. 样式：Select 下拉选择器，显示当前小区名称
3. 数据来源：调用 `getProjectList(companyId)` 获取该物业公司管理的小区列表
4. 切换行为：切换小区后，所有资产/收费/工单等数据自动刷新为当前小区数据
5. 存储：当前小区ID存入 `localStorage` 或 `Context`，页面刷新后保持
6. 权限：只有物业管理端显示此切换器，政府端/商家端/业主端不显示

## 六、数据库表结构建议

### 6.1 楼栋表（building）

```sql
CREATE TABLE building (
  id              INT PRIMARY KEY AUTO_INCREMENT,
  project_id      INT NOT NULL COMMENT '所属小区项目ID',
  name            VARCHAR(50) NOT NULL COMMENT '楼栋名称（如：1栋）',
  alias_name      VARCHAR(100) COMMENT '楼栋别名（如：紫荆苑）',
  total_layers    INT NOT NULL DEFAULT 1 COMMENT '总层数',
  underground_layers INT DEFAULT 0 COMMENT '地下层数',
  total_units     INT NOT NULL DEFAULT 1 COMMENT '单元数',
  total_elevators INT DEFAULT 0 COMMENT '电梯数',
  build_year      YEAR COMMENT '建成年份',
  property_type   VARCHAR(50) DEFAULT 'residence' COMMENT '房屋类型: residence/shop/office',
  data_source     VARCHAR(20) DEFAULT 'manual' COMMENT '数据来源: manual/gov_sync/import',
  sort_order      INT DEFAULT 0 COMMENT '排序号',
  enabled         TINYINT(1) DEFAULT 1,
  create_time     DATETIME DEFAULT CURRENT_TIMESTAMP,
  update_time     DATETIME DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP,
  KEY idx_project (project_id)
) COMMENT '楼栋表';
```

### 6.2 单元表（building_unit）

```sql
CREATE TABLE building_unit (
  id              INT PRIMARY KEY AUTO_INCREMENT,
  building_id     INT NOT NULL COMMENT '所属楼栋ID',
  name            VARCHAR(50) NOT NULL COMMENT '单元名称（如：1单元）',
  total_floors    INT DEFAULT 1 COMMENT '楼层数',
  total_houses    INT DEFAULT 0 COMMENT '房屋数',
  sort_order      INT DEFAULT 0,
  create_time     DATETIME DEFAULT CURRENT_TIMESTAMP,
  update_time     DATETIME DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP,
  KEY idx_building (building_id)
) COMMENT '单元表';
```

### 6.3 房屋表（house）

```sql
CREATE TABLE house (
  id                INT PRIMARY KEY AUTO_INCREMENT,
  project_id        INT NOT NULL COMMENT '所属小区项目ID',
  building_id       INT NOT NULL COMMENT '所属楼栋ID',
  unit_id           INT COMMENT '所属单元ID（无单元可为空）',
  floor             INT NOT NULL COMMENT '所在楼层',
  room_no           VARCHAR(50) NOT NULL COMMENT '房号（如：101）',
  full_name         VARCHAR(200) NOT NULL COMMENT '完整名称（如：1栋1单元101室）',
  layout            VARCHAR(50) COMMENT '户型（如：3室2厅1卫）',
  area              DECIMAL(10,2) COMMENT '建筑面积(㎡)',
  usable_area       DECIMAL(10,2) COMMENT '使用面积(㎡)',
  orientation       VARCHAR(20) COMMENT '朝向: east/south/west/north/south-north',
  decoration_status VARCHAR(20) DEFAULT 'rough' COMMENT '装修状态: rough/simple/standard/luxury',
  ownership_status  VARCHAR(20) DEFAULT 'vacant' COMMENT '产权状态: vacant/occupied/rented/for_sale/renovating',
  property_type     VARCHAR(50) DEFAULT 'residence' COMMENT '房屋类型: residence/shop/office/warehouse',
  data_source       VARCHAR(20) DEFAULT 'manual' COMMENT '数据来源: manual/gov_sync/import/owner_register',
  sort_order        INT DEFAULT 0,
  enabled           TINYINT(1) DEFAULT 1,
  create_time       DATETIME DEFAULT CURRENT_TIMESTAMP,
  update_time       DATETIME DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP,
  KEY idx_project (project_id),
  KEY idx_building (building_id),
  KEY idx_unit (unit_id),
  KEY idx_owner_status (ownership_status),
  UNIQUE KEY uk_building_room (building_id, unit_id, room_no)
) COMMENT '房屋表';
```

### 6.4 业主档案表（owner）

```sql
CREATE TABLE owner (

  native_place    VARCHAR(100) COMMENT '籍贯',

  profession      VARCHAR(100) COMMENT '职业',
  wechat_openid   VARCHAR(100) COMMENT '微信OpenID',
  tags            VARCHAR(500) COMMENT '标签（JSON数组，如：["党员","志愿者"]）',
  data_source     VARCHAR(20) DEFAULT 'manual' COMMENT '数据来源: manual/gov_sync/import/owner_register',
  status          TINYINT(1) DEFAULT 1,
  create_time     DATETIME DEFAULT CURRENT_TIMESTAMP,
  update_time     DATETIME DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP,

### 6.5 业主绑定表（house_owner）

```sql
CREATE TABLE house_owner (
  id              INT PRIMARY KEY AUTO_INCREMENT,
  house_id        INT NOT NULL COMMENT '房屋ID',
  owner_id        INT NOT NULL COMMENT '业主ID',
  owner_type      VARCHAR(20) NOT NULL DEFAULT 'owner' COMMENT '关系类型: owner/co_owner/family/tenant',
  bind_time       DATETIME NOT NULL COMMENT '绑定时间',
  unbind_time     DATETIME COMMENT '解绑时间',
  is_active       TINYINT(1) DEFAULT 1 COMMENT '是否当前生效',
  remark          VARCHAR(500) COMMENT '备注',
  create_time     DATETIME DEFAULT CURRENT_TIMESTAMP,
  KEY idx_house (house_id),
  KEY idx_owner (owner_id),
  KEY idx_active (is_active)

## 八、多小区切换上下文设计

### 8.1 设计思路

物业管理端需要支持多小区切换，核心设计如下：

```
┌─────────────────────────────────────────────────────────────────┐
│  CommunityProvider（React Context）                              │
│                                                                  │
│  interface CommunityContext {                                    │
│    currentCommunity: Organization | null;    // 当前选中小区      │
│    communityList: Organization[];            // 可管理的小区列表  │
│    switchCommunity: (id: number) => void;    // 切换小区          │
│    loading: boolean;                         // 加载状态          │
│  }                                                                 │
│                                                                  │
│  存储: localStorage 持久化 currentCommunityId                     │
│  初始化: 从 localStorage 读取 → 无则取第一个                      │
│  切换: 更新 Context → 更新 localStorage → 触发数据刷新            │
└─────────────────────────────────────────────────────────────────┘
```

### 8.2 影响范围

| 模块 | 影响 | 说明 |
|------|------|------|
| PCLayout Header | 新增小区切换器 Select | 仅物业管理端显示 |
| 资产总览 | 数据按 currentCommunity 筛选 | 所有统计卡片 |
| 楼栋管理 | 数据按 currentCommunity 筛选 | CRUD 操作 |
| 房屋管理 | 数据按 currentCommunity 筛选 | CRUD + 批量生成 |
| 业主管理 | 数据按 currentCommunity 筛选 | 业主列表 + 绑定 |
| 车位管理 | 数据按 currentCommunity 筛选 | CRUD + 绑定 |
| 收费管理（后续） | 数据按 currentCommunity 筛选 | 账单/收费 |
| 报修工单 | 数据按 currentCommunity 筛选 | 工单列表 |
| 仪表盘 | 数据按 currentCommunity 筛选 | 统计概览 |

### 8.3 实现步骤

1. 创建 [`CommunityContext`](property-management-system/src/contexts/CommunityContext.tsx) 提供者组件
2. 在 [`PCLayout.tsx`](property-management-system/src/layouts/PCLayout.tsx) 中包裹 `CommunityProvider`

4. 所有资产相关 Service 方法增加 `projectId` 参数
5. 页面组件通过 `useCommunity()` 获取当前小区 ID

## 九、与收费管理模块的衔接

房屋与车位管理是收费管理的前置基础模块，两者关系如下：

```
┌─────────────────────────────────────────────────────────────────┐
│  房屋、车位与业主管理模块（基础资产）                                 │
│                                                                  │
│  building(楼栋) → unit(单元) → house(房屋)                        │
│  parking_space(车位)                                              │
│  owner(业主档案) → house_owner(业主绑定)                           │
└──────────────────────────┬──────────────────────────────────────┘
                           │ 提供基础数据
                           ▼
┌─────────────────────────────────────────────────────────────────┐
│  收费管理模块（费用核算）                                          │
│                                                                  │
│  fee_item(费用项) → charge_rule(收费标准) → bill(账单)            │
│  ↑ 引用房屋的面积/类型/业主信息                                    │
│  ↑ 引用车位的类型/绑定业主信息                                     │
└─────────────────────────────────────────────────────────────────┘
```

**具体衔接点：**

1. **账单生成时**：遍历所有房屋 → 根据房屋类型匹配收费标准 → 按面积/固定金额计算 → 生成账单 → 推送给绑定业主
2. **车位收费**：遍历所有固定车位 → 根据车位类型匹配收费标准 → 生成车位费账单 → 推送给绑定业主
3. **业主端缴费**：业主登录后 → 查询名下所有房屋和车位的待缴账单 → 合并展示 → 一键缴费
