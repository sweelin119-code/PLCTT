## 一、整体架构

### 1.1 统一用户中心

```
┌──────────────────────────────────────────────────────────┐
│                    统一用户认证中心                        │
│              手机号 + 密码 / 微信授权登录                   │
├──────────┬──────────┬──────────┬──────────┬──────────────┤
│ 政府监管端 │ 物业管理端 │ 商家管理端 │ 业主微信端 │ 公众号管理    │
│ 市/区/街道 │ 公司/项目  │ 店铺/店员  │ 业主/家属  │ 运营人员      │
└──────────┴──────────┴──────────┴──────────┴──────────────┘
```

### 1.2 核心原则

| 原则 | 说明 |
|------|------|
| **一个用户，多端角色** | 同一手机号可在不同端口拥有不同角色（如：物业经理 + 业主） |
| **数据权限隔离** | 上级可看下级数据，同级互不可见 |
| **组织树驱动权限** | 用户权限由「所属组织 + 角色」共同决定 |
| **事件驱动流转** | 跨角色协作通过工单/事件流转实现 |

## 三、数据权限规则

### 3.1 数据可见范围矩阵

以「投诉管理」为例，不同角色看到的数据范围：

| 角色 | 数据范围规则 | 可见数据 |
|------|-------------|---------|
| 市级管理员 | 全市所有组织 | 杭州市所有投诉 |
| 区级管理员 | 本区及下级街道 | 西湖区 + 下属街道投诉 |
| 街道办人员 | 本街道及下属小区 | 文新街道 + 下属小区投诉 |
| 公司管理员 | 本公司所有项目 | 公司A所有小区投诉 |
| 项目经理 | 本小区 | 桂花城小区投诉 |
| 客服人员 | 本小区 | 桂花城小区投诉 |
| 业主 | 本人 | 自己的投诉记录 |

### 3.2 数据权限实现逻辑

```
用户登录 → 获取用户角色列表 → 获取每个角色关联的组织ID
         → 根据组织树计算可见范围（本级及下级）
         → SQL注入组织范围条件
```

## 四、物业管理端 — 人员管理功能设计

### 4.1 页面规划

| 页面 | 路由 | 说明 |
|------|------|------|
| 人员列表 | /property/staff/list | 查看/搜索/筛选公司所有人员 |
| 新增人员 | /property/staff/add | 添加新用户并分配角色 |
| 编辑人员 | /property/staff/edit/:id | 修改人员信息/角色 |
| 角色管理 | /property/roles | 查看公司内角色分配情况 |

### 4.2 人员列表页面

**功能点：**
- 搜索：姓名、手机号模糊搜索
- 筛选：按角色（项目经理/客服/工程等）、按项目（小区）
- 列表字段：姓名、手机号、角色、所属项目、状态、操作
- 操作：编辑、禁用/启用、重置密码

### 4.3 新增/编辑人员

**表单字段：**
- 姓名（必填）
- 手机号（必填，唯一校验）
- 角色（必选，可多选？还是单选？）
- 所属项目（根据角色自动关联）
- 状态（默认启用）

### 4.4 角色管理页面

**功能点：**
- 查看公司内各角色的用户列表
- 了解每个角色下有哪些人
- 支持从角色维度管理人员

## 五、登录流程设计

### 5.1 PC端登录（政府/物业/商家/公众号）

```
用户打开页面 → 进入登录页 → 输入手机号+密码
       → 后端验证 → 返回JWT Token + 用户信息 + 角色列表
       → 前端根据角色列表跳转对应端口首页
```

### 5.2 业主微信端登录

```
用户打开微信H5 → 微信OAuth授权 → 获取OpenID
       → 判断是否已绑定手机号
       → 已绑定 → 自动登录
       → 未绑定 → 跳转手机号绑定页 → 绑定后登录
```

### 5.3 多角色用户登录

```
用户登录 → 后端返回该用户的所有角色列表
       → 如果只有1个角色 → 直接进入对应端口
       → 如果有多个角色 → 展示角色选择页
       → 用户选择本次要使用的角色 → 进入对应端口
```

## 七、实施计划

| 阶段 | 内容 | 优先级 |
|------|------|--------|
| 第一阶段 | 物业管理端 → 人员管理（CRUD）+ 角色管理 | **最高** |
| 第二阶段 | 登录页 + 统一认证 + JWT + 路由守卫 | 高 |
| 第三阶段 | 政府端 → 人员管理 + 组织树 | 中 |
| 第四阶段 | 商家端 → 人员管理 | 中 |
| 第五阶段 | 业主微信端 → 自注册 + 物业审核 | 中 |

## 八、确认结论

| # | 问题 | 结论 |
|---|------|------|
| 1 | 物业端人员管理中，一个人是否可以同时拥有多个角色？ | ✅ **允许**，一个人可拥有多个角色 |
| 2 | 是否需要「角色管理」页面来配置每个角色的权限？ | ✅ **需要**，开发角色管理页面，支持配置角色权限 |
| 3 | 密码规则 | ✅ **手机号和密码默认为一样**，登录后可在个人中心修改 |
| 4 | 验证码登录 | ✅ **引入验证码**，但不接短信平台，前端随机生成6位数字验证码（模拟验证码功能） |

## 九、开发计划（第一阶段）

### 9.1 需要开发的页面

| # | 页面 | 路由 | 说明 |
|---|------|------|------|
| 1 | 登录页 | /login | 手机号+密码登录，支持验证码模式 |
| 2 | 物业管理端-人员列表 | /property/staff | 查看/搜索/筛选/管理公司人员 |
| 3 | 物业管理端-新增人员 | /property/staff/add | 添加用户+分配角色+关联项目 |
| 4 | 物业管理端-编辑人员 | /property/staff/edit/:id | 修改信息/角色 |
| 5 | 物业管理端-角色管理 | /property/roles | 角色列表+权限配置 |
| 6 | 物业管理端-角色新增/编辑 | /property/roles/add | 创建角色+分配权限 |

### 9.2 技术实现方案

由于当前是纯前端项目（Create React App），第一阶段采用 **Mock数据 + localStorage 模拟后端** 的方式实现：

1. 在 `src/services/` 下创建 mock 数据层
2. 使用 React Context 实现全局用户状态管理
3. 路由守卫：未登录跳转登录页
4. 数据权限：通过当前用户角色+组织ID过滤数据
